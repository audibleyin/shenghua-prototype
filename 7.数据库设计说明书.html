1 引言

1.1 编写目的

本文档为生花AI创意平台数据库设计说明书，旨在为系统数据库实现提供详细指导。本文档面向数据库管理员、系统架构师、后端开发工程师及测试人员，详细阐述数据库的逻辑结构、物理设计、对象设计及运维策略，作为数据库开发与维护的技术规范依据。

2 项目概述

生花AI创意平台是一个连接AI漫剧创作人才与项目需求的交易撮合平台，核心功能包括：双端身份体系、接单大厅、人才信息库、创作广场、数字资产库、培训学院、L1-L5等级体系、社区互动及平台活动对接。

数据库关键性能指标：月活跃用户20,000+，月度GMV 500万，月度订单数2,000+，创作者规模15,000+，广场作品数10,000+，资产库规模20,000+，系统并发500 QPS。

3 数据库外部环境及资源规划

3.1 数据库环境要求

硬件要求：CPU 8核，内存32GB，存储1TB NVMe SSD，网络千兆以太网。

软件要求：MySQL 8.0+（InnoDB引擎，utf8mb4字符集），Redis 7.0+，CentOS/Ubuntu LTS。

性能指标：简单查询≤50ms，复杂查询≤500ms，最大并发500连接，可用性99.9%。

3.2 存储资源规划

存储需求：初始500GB SSD，年增长预留50%。用户相关10GB，项目订单50GB，内容创作100GB，资产库30GB，培训社区20GB，日志100GB。

架构设计：MySQL主库(8C/32G) + 从库(8C/32G) + Redis缓存(4C/16G)。对象存储存放文件，ELK存放日志。

归档策略：交易明细保留3年，日志保留6个月归档。

4 表结构设计

4.1 关系数据库表结构

数据库采用MySQL 8.0 InnoDB引擎，默认字符集utf8mb4，排序规则utf8mb4_unicode_ci。

命名规范：
· 表名：小写字母+下划线，如 user、creator_profile
· 字段名：小写字母+下划线，如 user_id、created_at
· 主键：统一使用 id（BIGINT自增）
· 外键：字段名以 _id 结尾
· 索引：idx_ 前缀，唯一索引 uk_ 前缀
· 时间字段：统一使用 datetime
· 软删除：统一使用 is_deleted（tinyint, 0=正常, 1=删除）

4.3 关系说明

核心数据流：用户注册后创建user + user_identity。创作者入驻：creator_profile -> creator_review审核 -> talent_info入库 -> 获得等级。项目交易：client发布project -> creator投标project_bid -> 中标生成order -> 里程碑交付 -> review评价 -> transaction流水。内容创作：work发布 -> 版权交易copyright_transaction。资产使用：asset上架 -> asset_usage记录 -> 分润。

外键关系汇总：user_identity.user_id -> user.id；user_auth.user_id -> user.id；creator_profile.user_id -> user.id；creator_profile.professional_role_id -> professional_role.id；talent_info.user_id -> user.id；project.client_id -> user.id；project_bid.project_id -> project.id；project_bid.creator_id -> user.id；order.project_id -> project.id；order.client_id/creator_id -> user.id；order_milestone.order_id -> order.id；work.creator_id -> user.id；asset.uploader_id -> user.id；asset.category_id -> asset_category.id；asset_usage.asset_id -> asset.id；course_enrollment.user_id -> user.id；course_enrollment.course_id -> course.id；review.order_id -> order.id；review.from_id/to_id -> user.id；membership.user_id -> user.id。

5 数据库对象设计

5.1 视图设计

v_creator_dashboard：创作者工作台视图，关联creator_profile + order + review。
v_talent_search：人才搜索视图，关联talent_info + professional_role + creator_level。
v_project_detail：项目详情视图，关联project + user + project_bid。
v_monthly_settlement：月度结算视图，关联order + transaction + membership。

5.2 存储过程设计

sp_calculate_commission：根据订单金额、创作者等级、会员身份计算实际佣金和各方分润。
sp_monthly_membership_charge：月度会员费核算，按"先享后付"规则判断是否扣费。
sp_auto_level_upgrade：每月1日自动扫描创作者等级变更，满足条件的自动升级/降级。
sp_tuition_refund：订单完成后自动触发学费返还计算。

5.3 触发器设计

trg_order_completed：订单完成后自动创建transaction流水、更新creator_profile统计数据。
trg_bid_accepted：投标中标后自动更新project状态为in_progress，拒绝其他投标。
trg_asset_used：资产使用后自动递增asset.usage_count，计算uploader_share。

6 数据库运维设计

6.1 数据恢复策略

备份策略：每日全量备份（凌晨2:00），每小时增量备份。全量备份保留30天，增量保留7天。
恢复策略：任意时间点恢复(PITR)，基于全量+binlog实现。RTO<=2小时，RPO<=15分钟。
灾备方案：主从架构，从库异地备份。主库故障时自动切换。

6.2 性能优化策略

SQL优化：所有查询使用索引避免全表扫描；复杂查询限制行数使用分页；定期分析慢查询日志；避免WHERE字段函数运算。

数据库参数：innodb_buffer_pool_size=物理内存70%；innodb_log_file_size=2GB；max_connections=500；long_query_time=1s。

索引维护：定期检查索引碎片，OPTIMIZE TABLE重建。每月分析索引使用频率，清理无效索引。
